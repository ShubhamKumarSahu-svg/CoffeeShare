# Chapter 7: Database Management System (DBMS) Architecture

CoffeeShare is fundamentally a serverless, peer-to-peer application. So why does it need a database at all? 

This document explores the exact role of the Database Management System (DBMS), the schema design, and why we chose **Supabase (PostgreSQL)** over alternatives like MongoDB or Firebase.

## 7.1 The Role of the DBMS in a Serverless App

In a P2P application, the computers eventually talk directly to each other. However, computers cannot magically find each other on the internet. They require a "Signaling Server" to act as a centralized matchmaker.

The sole responsibility of the DBMS in CoffeeShare is **Signaling and State Management for Room Creation**.
1. **Host generates a Session**: The Uploader creates a random 6-character shortcode (e.g., `A4X9B2`).
2. **Database Write**: The Uploader inserts a row into the database linking `A4X9B2` to their massive WebRTC `peerId` (a long UUID generated by PeerJS).
3. **Database Read**: The Downloader types `A4X9B2`, queries the database, and retrieves the Uploader's `peerId`.
4. **Disconnection**: Once the WebRTC handshake completes, the database is abandoned. No files or chat logs are *ever* saved to the database.

## 7.2 Why PostgreSQL / Supabase?

We selected Supabase (built on top of PostgreSQL) for several critical reasons:

1. **Relational Constraints & ACID Properties**: Even though our schema is simple, PostgreSQL provides strict ACID compliance. If two users try to claim the exact same 6-character shortcode at the exact same millisecond, PostgreSQL's primary key constraints prevent a race condition, safely rejecting the second claim.
2. **Realtime Capabilities**: Supabase features a native `PostgreSQL Realtime` engine. It listens to PostgreSQL's logical replication stream and broadcasts `INSERT`, `UPDATE`, and `DELETE` events over WebSockets. This means if we wanted to build a "Waiting for Downloader..." UI, the host's browser could subscribe to the row and instantly react when a downloader connects, without needing long-polling.
3. **Row Level Security (RLS)**: PostgreSQL allows us to secure the data at the database layer. We can write SQL policies that dictate exactly who is allowed to read or write specific rows, ensuring malicious actors cannot scrape active session IDs.

## 7.3 Schema Design

The schema is intentionally minimalist to ensure ultra-fast reads and writes, as the signaling phase must complete in milliseconds.

**Table: `channels`**

| Column Name | Data Type   | Constraints                  | Description                                                                 |
| :---        | :---        | :---                         | :---                                                                        |
| `id`        | `uuid`      | `PRIMARY KEY`, `DEFAULT gen_random_uuid()` | The unique identifier for the database row.                                 |
| `slug`      | `text`      | `UNIQUE`, `NOT NULL`         | The 6-character human-readable shortcode (e.g., `A4X9B2`).                  |
| `peer_id`   | `text`      | `NOT NULL`                   | The UUID generated by PeerJS, required to initiate the WebRTC handshake.    |
| `created_at`| `timestamp` | `DEFAULT now()`              | Used for garbage collection and TTL (Time-To-Live).                         |

### The TTL (Time-To-Live) Strategy
Because CoffeeShare creates ephemeral rooms, the database would eventually fill up with dead `slug` to `peer_id` mappings. 
To prevent database bloat:
1. **Application-Level Deletion**: When the Uploader closes their tab, a `beforeunload` event triggers an API call or Supabase client call to `DELETE FROM channels WHERE peer_id = '...'`.
2. **Database-Level CRON**: A PostgreSQL `pg_cron` job can be scheduled to automatically execute `DELETE FROM channels WHERE created_at < NOW() - INTERVAL '24 hours'`, ensuring perfect database hygiene without manual intervention.

## 7.4 Security & Privacy

The DBMS is the *only* part of CoffeeShare that retains state on a centralized server. Therefore, its security model is paramount.
*   **Zero Knowledge**: The database has zero knowledge of the files being transferred. It does not store file names, sizes, binary blobs, or IP addresses.
*   **Ephemeral Data**: The `peer_id` is a randomly generated UUID that changes every time the user refreshes the page. It cannot be tied back to a physical user or device.
*   Once the P2P connection drops, the room is destroyed. Even if the Supabase instance was compromised, the attacker would only find a list of meaningless, expired UUIDs.
